# 专业领域术语管理（4 领域）

## 📚 专业领域术语管理

> 不同专业领域的术语管理方式截然不同。翻译时必须在领域内建立术语体系，保持全篇一致性。

### 通用术语管理原则

1. **一术语一译名**：同一篇翻译中，同一个源语术语只能有一个译名
2. **优先权威来源**：有国家标准/行业标准的术语，优先采标
3. **首次出现标注原文**：专业术语首次出现时，在译文后附原文（括号内）
4. **建立术语表**：长文翻译前先提取术语表，统一译法后再开始翻译
5. **联网查证**：不确定的术语先搜索，不猜测

### 领域1：法律翻译术语管理

#### 核心特征
- 术语有严格的法律定义，一个词的选错可能导致法律后果
- 同一术语在不同法系（大陆法系 vs 英美法系）中含义可能不同
- 法律术语通常不可替换、不可缩写、不可意译

#### 术语管理策略

| 策略 | 操作 | 示例 |
|------|------|------|
| **定义驱动** | 先理解法律定义，再确定译名 | "consideration"不是"考虑"而是"对价"（合同法核心概念） |
| **法系标注** | 标注术语所属法系 | "衡平法上的……"（英美法系特有概念） |
| **原文保留** | 关键法律概念首次出现保留原文 | "不可抗力（Force Majeure）" |
| **力度精确** | shall/must/may/will 的翻译力度必须精确对应 | shall=应当/须（义务）、may=可（权利/选择）、must=必须（强制） |

#### 高频法律术语对照

| 英文 | 中文 | 备注 |
|------|------|------|
| Force Majeure | 不可抗力 | 大陆法系和英美法系定义有差异 |
| Indemnification | 赔偿/补偿/免责 | 需结合上下文判断具体类型 |
| Warranty vs Representation | 保证 vs 陈述 | 法律后果不同，不可混用 |
| Liquidated Damages | 约定损害赔偿金 | 不等于"违约金"（两者法律性质不同） |
| Governing Law | 管辖法律 | ≠"适用法律"（Governing Law有特定含义） |
| Without Prejudice | 不损害权利/不妨害 | 法律特权概念，不可省略 |
| Material Adverse Change | 重大不利变更 | M&A并购交易中的核心条款术语 |
| Severability | 可分割性 | 合同条款效力术语 |

#### 术语自检清单
- □ 所有shall/must/may的翻译力度是否准确？
- □ 同一法律概念全篇是否使用相同译名？
- □ 大陆法系和英美法系的术语是否做了区分标注？
- □ 法律后果不同的术语（如warranty vs representation）是否做了区分？

### 领域2：医学翻译术语管理

#### 核心特征
- 医学术语高度专业化，英文大量使用希腊/拉丁词根
- 一个术语可能对应多个中文译名（大陆 vs 台湾 vs 香港）
- 药品名称有通用名（INN）和商品名之分
- 解剖学术语有严格的命名体系

#### 术语管理策略

| 策略 | 操作 | 示例 |
|------|------|------|
| **词根拆解** | 英文医学术语可从词根推导含义 | "cholecystectomy" = chole(胆) + cyst(囊) + ectomy(切除术) = 胆囊切除术 |
| **标准采标** | 优先采用全国科学技术名词审定委员会的医学名词 | "Alzheimer's disease" → "阿尔茨海默病"（而非"老年痴呆症"，后者不准确且有歧视性） |
| **区域适配** | 根据目标受众选择大陆/台湾/香港译名 | "Antibiotic" → 大陆：抗生素 / 台湾：抗生素/抗菌素 |
| **药品名规范** | 通用名用INN中文译名，商品名保留原文+® | "Aspirin" → 通用名：阿司匹林；商品名："拜阿司匹林®（Bayer Aspirin）" |

#### 高频医学术语对照

| 英文 | 大陆 | 台湾 | 备注 |
|------|------|------|------|
| Alzheimer's Disease | 阿尔茨海默病 | 阿茲海默症 | 避免使用"老年痴呆症" |
| Parkinson's Disease | 帕金森病 | 巴金森氏症 | |
| Antibiotic | 抗生素 | 抗生素/抗菌素 | 两岸基本一致 |
| CT Scan | CT检查/计算机断层扫描 | 電腦斷層掃描 | 科普文可用"CT"，学术文用全称 |
| Informed Consent | 知情同意 | 知情同意 | 法律医学交叉术语，意译一致性重要 |
| Placebo | 安慰剂 | 安慰劑 | |
| Metastasis | 转移 | 轉移 | |

#### 术语自检清单
- □ 医学术语是否查证过权威来源（而非依赖通用翻译记忆）？
- □ 药品名称是否区分了通用名和商品名？
- □ 解剖学术语是否使用了标准命名？
- □ 目标受众是专业人士还是普通读者（影响术语解释策略）？

### 领域3：金融翻译术语管理

#### 核心特征
- 金融术语存在大量"假朋友"——看起来像日常英语但有特殊金融含义
- 不同市场（A股/港股/美股）的同一概念可能有不同术语
- 数字格式和货币单位需要精确处理
- 监管术语在不同司法管辖区含义不同

#### 术语管理策略

| 策略 | 操作 | 示例 |
|------|------|------|
| **市场适配** | 对应用户所在市场选择术语 | "IPO" → 中文"首次公开募股"或直接用"IPO"（根据受众） |
| **假朋友识别** | 识别日常词在金融语境中的特殊含义 | "security" → 不是"安全"而是"证券"；"position" → 不是"位置"而是"仓位/头寸" |
| **数字规范** | 大数字按目标市场习惯格式化 | 1,000,000 → 100万（中文）、1 million（英文） |
| **监管区分** | 标注术语所属监管体系 | "招股说明书"在A股/港股/美股中的对应文件不同 |

#### 高频金融术语对照

| 英文 | 中文（大陆） | 中文（香港） | 备注 |
|------|------------|------------|------|
| IPO | 首次公开募股/IPO | 新股上市/IPO | 业内通常直接用IPO |
| EPS | 每股收益 | 每股盈利 | |
| Due Diligence | 尽职调查 | 盡職審查 | |
| Hedge | 对冲 | 對沖 | |
| Bond vs Debenture | 债券（通用）vs 信用债券 | 債券 vs 債權證 | 两个词有法律区别 |
| Default | 违约 | 違約 | |
| M&A | 并购 | 併購 | Mergers & Acquisitions |
| Bull/Bear Market | 牛市/熊市 | 牛市/熊市 | 两岸一致 |
| Custodian | 托管人 | 託管人 | 资产托管业务核心术语 |

#### 术语自检清单
- □ 是否检查了"假朋友"词（security/position/instrument/equity等）？
- □ 数字格式和单位是否按目标市场习惯做了转换？
- □ 监管文件术语是否匹配正确的司法管辖区？
- □ 货币代码和金额格式是否正确？

### 领域4：IT/技术翻译术语管理

#### 核心特征
- 大量术语不翻译（保留英文原文）——代码/命令/API名称/产品名
- 技术社区中存在约定俗成的译法，不可自创
- 大厂（微软/Google/Apple）对同一概念可能有不同官方中文译名
- 新兴技术术语的中文译名可能尚未标准化

#### 术语管理策略

| 策略 | 操作 | 示例 |
|------|------|------|
| **保留不译清单** | 明确哪些必须保留原文 | 代码/变量名/函数名/命令行/API端点/文件名 → 永不翻译 |
| **社区采标** | 技术术语优先采用社区约定译法 | "container" → "容器"（已约定俗成）而非"集装箱" |
| **大厂词典** | 翻译某个平台/产品时，使用该厂商的官方中文术语 | Microsoft的"sign in" → "登录"；Google的"sign in" → "登录"（微妙差异需查官方文档） |
| **首次标注** | 不确定社区是否认可时，首次标注原文 | "微服务（Microservices）" |
| **版本敏感** | 技术概念随版本变化，翻译需对应版本语境 | Kubernetes v1.24前有Dockershim，之后移除——相关翻译需注意语境 |

#### 高频IT术语对照

| 英文 | 中文 | 保留英文？ | 备注 |
|------|------|-----------|------|
| API | API/应用程序接口 | 多数情况保留API | 专业读者直接说API |
| Container | 容器 | 翻译 | Docker/K8s语境中已标准化 |
| Deploy/Deployment | 部署 | 翻译 | |
| Microservices | 微服务 | 翻译 | |
| Cache | 缓存 | 翻译 | |
| Load Balancer | 负载均衡器/负载均衡 | 翻译 | |
| Firewall | 防火墙 | 翻译 | |
| Cloud Native | 云原生 | 翻译 | |
| CI/CD | CI/CD | 保留 | 行业通用缩写 |
| Git/commit/push/pull | Git/commit/push/pull | 全部保留 | Git命令永远不翻译 |
| Agile/Scrum | 敏捷/Scrum | 混合 | Agile翻译，Scrum保留 |
| Machine Learning | 机器学习（ML） | 翻译+缩写 | |
| Open Source | 开源 | 翻译 | |

#### 不翻译清单（IT领域）

以下类别在翻译时**始终保留原文**：
1. **代码**：所有编程语言的代码块、变量名、函数名、类名
2. **命令行**：所有CLI命令、参数、flag
3. **API名称**：REST端点、GraphQL查询名、SDK方法名
4. **文件名/路径**：所有文件路径和文件名
5. **产品名/服务名**：AWS Lambda、Google Cloud Run、Azure Functions → 保留原名
6. **缩写/协议名**：HTTP、HTTPS、TCP/IP、JSON、YAML → 保留原名
7. **错误信息**：系统错误信息保留原文+中文解释

#### 术语自检清单
- □ 代码/命令/API是否保留了原文？
- □ 技术术语是否参考了社区约定译法而非自创？
- □ 官方文档术语是否与该厂商保持一致？
- □ 新兴技术术语是否加注了原文？

### 跨领域术语管理总表

| 管理维度 | 法律 | 医学 | 金融 | IT |
|----------|------|------|------|-----|
| **权威来源** | 法条原文+法学词典 | CNCTST+MeSH+WHO | 监管文件+行业标准 | 官方文档+社区 |
| **保留原文频率** | 高（首次标注） | 中 | 低-中 | 极高 |
| **区域差异** | 法系差异大 | 大陆/台湾/香港 | A股/港股/美股 | Apple/Google/Microsoft |
| **翻译自由度** | 极低，不可自由发挥 | 低，需权威采标 | 中，有专业约定 | 中-高，社区驱动 |
| **最危险错误** | 法律后果 | 医疗事故风险 | 财务损失 | 系统故障 |
| **诚实标注策略** | 不确定必须标注+建议咨询律师 | 不确定必须标注+建议咨询医生 | 不确定标注+注明数据源 | 不确定标注+搜索社区 |

### 术语管理工具建议

在实际翻译中，可采用以下方法建立术语一致性：

1. **术语提取**：翻译长文前，先扫描全文提取10-30个关键术语
2. **术语卡**：为每个关键术语建立术语卡，记录"源语 → 目标语 + 领域 + 选择理由 + 参考来源"
3. **搜索交叉验证**：重要术语搜索≥2个权威来源确认
4. **术语一致性自检**：翻译完成后，用搜索功能检查同一源语术语是否出现了不同译名
5. **领域标注**：在翻译说明中标注本翻译使用的术语参考来源
