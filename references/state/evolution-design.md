# 技能自进化设计（架构自知 / 进化决策 / 进化执行 / 进化治理 / 元进化）

## 🧬 技能自进化设计

本技能的进化体系分为五层。每一层只依赖下一层提供的信息，不跨层跳。

---

### 第一层：架构自知（Architecture Self-Awareness）

> 要重构逻辑，首先得说清楚"现在的逻辑是什么"。

**组件清单与核心假设：**

```
组件清单
│
├── 场景路由（S1-S8）
│   ├── 依赖：触发词表 + 语言识别
│   ├── 核心假设：8个场景覆盖主流翻译需求
│   ├── 场景参数：风格基调 + 策略倾向 + 典型处理 + 场景红线
│   └── 易变点：新场景（如游戏本地化/影视译制）需要扩展
│
├── 五步翻译法
│   ├── 步骤：理解原文→确定场景受众→选择策略→逐层翻译→质量自检
│   ├── 核心假设：翻译质量可以通过这五个可独立优化的步骤保证
│   ├── 步骤间关系：前一步的输出是后一步的输入，线性递进
│   └── 易变点：步骤顺序是否需要调整？是否需要增加"译者笔记"步骤？
│
├── 四层翻译深度（L1-L4）
│   ├── L1信息准确 → L2表达通顺 → L3场景地道 → L4风格匹配
│   ├── 核心假设：翻译质量可拆分为这四个递进层次
│   ├── 层级关系：必须逐层完成，不可跳跃
│   └── 易变点：层数需要增加？层级间是否有重叠可合并？
│
├── 三策略体系
│   ├── 直译 / 意译 / 本地化
│   ├── 核心假设：三种策略可覆盖所有翻译需求的策略选择
│   ├── 策略边界：直译↔意译之间是否有灰色地带？
│   └── 易变点：是否需要第四种策略（如"创造性翻译"独立出来）？
│
├── 本地化适配
│   ├── 文化适配 + 单位换算 + 称呼转换
│   ├── 核心假设：语言翻译之外的文化要素可从这三方面系统处理
│   └── 易变点：适配维度需要扩展（如视觉元素适配、平台适配）？
│
├── 六维质量评分
│   ├── 准确/通顺/地道/风格/实用/诚实
│   ├── 核心假设：翻译输出质量可被这六个维度充分衡量
│   └── 易变点：评分维度的完备性和权重
│
└── 输出格式
    ├── 标准版 / 简版 / 校对版 / 多语种对比 四模式
    └── 核心假设：四种输出模式满足不同复杂度需求
```

### 第二层：进化决策（Evolution Decision Framework）

> 不是每次不满意都要改。先分级，再行动。

#### 2.1 问题分级

```
┌─────────────────────────────────────────────────────┐
│ L0 噪音                                              │
│ 偶发的单次翻译失误。不构成模式。                        │
│ → 记录日志，不做任何改动。                            │
├─────────────────────────────────────────────────────┤
│ L1 案例缺口                                          │
│ 反复出现的翻译问题类型，当前框架能诊断但缺乏处理指南。    │
│ 例如：某个新兴网络用语的翻译每次都需临时判断。           │
│ → 新增翻译案例条目，不动框架。                        │
├─────────────────────────────────────────────────────┤
│ L2 框架微调                                          │
│ 某个翻译步骤需要补充子规则、调整顺序、细化标准。         │
│ 核心逻辑不变。                                       │
│ → 走轻量验证（7天/30次翻译，单轨对照即可）。            │
├─────────────────────────────────────────────────────┤
│ L3 框架重构                                          │
│ 翻译流程、层级结构、策略体系需要改。核心假设变了。        │
│ → 走完整双轨验证（最少50次翻译，最少7天）。              │
├─────────────────────────────────────────────────────┤
│ L4 范式替换                                          │
│ 整套翻译方法论被更优方案替代。                          │
│ → 走完整双轨 + 外评估（专业译者评审/用户反馈）。         │
│ → 旧范式存档到"历史范式库"，永久保留。                 │
└─────────────────────────────────────────────────────┘
```

#### 2.2 进化前判决清单

在启动任何 L2 及以上改动之前，必须先过这五道闸门：

```
① 穷尽了吗？
   在现有框架内调整参数（策略选择标准、L层级判断）能解决问题吗？
   如果能 → 不启动重构。

② 是框架问题还是数据问题？
   翻译失效是因为方法论不对，还是因为遇到的文本超出了框架设计范围？
   如果是后者 → 先考虑扩展场景，再考虑重构。

③ 改了之后影响面多大？
   这个组件被哪些其他组件依赖？（查架构清单，第一层）
   改动会连锁影响什么？

④ 有更简单的解决方案吗？
   如果没有独立证据说明新逻辑更好 → 不要动。
   "感觉不够好"不是重构的理由。

⑤ 现在是最佳时机吗？
   技能是否处于稳定使用期？用户是否依赖当前翻译输出格式？
   如果正在被高频使用 → 推迟，或先在沙箱中验证。
```

#### 2.3 不该进化的场景

明确列出"不要动"的情况：

- ❌ 用户满意度高且没有新的失效信号 → **稳定 > 完美**
- ❌ 仅为了"更优雅"而重构 → **拒绝**
- ❌ 没有明确的对比验证方案 → **拒绝**
- ❌ 上一次重构还在验证期内 → **排队，不叠加重构**

### 第三层：进化执行（Evolution Execution）

> L1 案例新增直接更新。L2-L4 走下面的完整流程。

#### 3.1 进化沙箱

L3 及以上改动，先在沙箱中验证：

```
沙箱条件：
- 准备50-100条涵盖不同场景、不同语言对、不同难度的翻译测试用例
- 新旧框架各自翻译一遍
- 对比维度：
  ├─ 准确度：错误率对比
  ├─ 通顺度：翻译腔出现频率
  ├─ 地道度：母语者盲评可接受率
  ├─ 翻译效率：平均每千字的处理时间/步骤数
  └─ 用户可接受率：不用新旧标注，随机展示给测评者
- 沙箱通过 → 进入双轨在线验证
- 沙箱不通过 → 修改候选框架，不进入在线阶段
```

#### 3.2 双轨在线验证

```
双轨验证协议
│
├─ ① 设定验证期
│     最少50次翻译，或最少7天，取长者。
│     覆盖不同场景和语言对——不能全是一种场景。
│
├─ ② 每次翻译双轨输出
│     同一条输入，旧框架和新框架各自翻译一遍。
│     记录：哪条轨道的翻译更准确？更通顺？更地道？
│     记录：两条轨道译法不同的关键决策点（最有价值的对比数据）。
│
├─ ③ 对比维度
│     ├─ 准确率：信息遗漏/增补/歪曲的频率
│     ├─ 通顺度：翻译腔出现频率
│     ├─ 地道度：是否像目标语原创
│     ├─ 效率：是否需要更多步骤或时间
│     └─ 用户接受度：如果用户在交互中给了反馈，记录下来
│
├─ ④ 分歧裁决
│     两轨译法不同时，不自动选新或旧。
│     人工判断哪个译法更好。
│     裁决结果同时记录——这对评估两套逻辑很重要。
│
└─ ⑤ 渐变切换（仅L3/L4）
      验证期结束后，如果新框架胜出：
      - 第1-3天：新框架处理 30% 的翻译请求（随机抽样）
      - 第4-7天：新框架处理 60%
      - 第8天起：新框架处理 100%
      - 任何一个阶段出现显著恶化 → 暂停、分析、决定
```

#### 3.3 回滚协议

```
触发回滚的条件（满足任一）：
- 双轨验证中新框架的翻译准确率 < 旧框架
- 用户明确对新框架翻译表示不满意 ≥3 次（在验证期内）
- 新框架的翻译腔出现率 > 旧框架的 1.5 倍
- 新框架的术语不一致问题 > 旧框架的 2 倍

回滚操作：
- 恢复旧框架为主逻辑
- 新框架存档到"历史演化记录"，标注"验证未通过"
- 记录回滚原因和对比数据
- 版本号不递增（这次重构视为未发生）
```

#### 3.4 废弃/融合

```
验证结论 → 新框架更优
    │
    ├─ ① 保留旧框架作为"vX 历史版本"记录在技能末尾
    │     不是删掉，是存档。
    ├─ ② 新框架替换为主逻辑
    │     版本号按变更规模递增
    ├─ ③ 保留旧版本至少一个大版本周期
    └─ ④ 记录重构日志（为什么换、换了什么、验证数据摘要）

验证结论 → 各有优劣
    │
    └─ 融合方案：取出各自的长处，拼成新版本

验证结论 → 新框架更差
    │
    └─ 放弃方案：记录"什么情况下旧框架比新框架好"
       这个记录本身就是重要的翻译认知。
```

### 第四层：进化治理（Evolution Governance）

#### 4.1 版本体系

```
v[major].[minor].[patch]

patch: L1 案例新增、术语表更新、措辞调整
minor: L2 框架微调（新增场景、调整L层级标准、优化策略选择逻辑）
major: L3/L4 框架重构、范式替换

版本记录格式：
---
version: 1.1.6
date: 2026-07-07
type: L1 结构升级（skill-maker-pro v3.0.0 双层拆分）
change: 单文件 99.8KB 拆分为外壳(≤30KB)+references/ 双层结构；S1-S8 细则/五步法/四层深度/四把诊断刀/12 组文化陷阱/4 领域术语/完整示例/自进化设计迁移至 references/；第1行焊入起源铭文；触发词原声化；evolution.tier 写入 YAML
reason: 过去30天用户请求中游戏翻译占比12%，当前8场景无法覆盖
sandbox: 40条游戏翻译测试，新场景处理准确率94%
dual_track: 7天/35次翻译，新场景处理效果良好
user_feedback: 无负面
---
```

#### 4.2 进化日志（Evolution Log）

```
每次进化尝试（无论成功还是失败）必须记录：

- 触发原因：什么信号触发了进化
- 尝试的方案：怎么改的
- 验证方法：沙箱+双轨，还是仅双轨
- 结果：通过/失败/部分通过
- 关键教训：这次进化教会了我们什么
- 对进化协议的影响：协议本身需要调整吗？

目的：让进化协议越来越了解"什么样的进化会成功"。
```

#### 4.3 跨技能进化遗传

```
当 translate-pro 完成一次 L3 重构后：
→ 提炼"重构模式"（如：场景拆分模式、层级重组模式）
→ 存入元技能（"制作技能的技能"）的"进化模式库"
→ 后续技能在遇到类似问题时，直接参考这个模式
```

### 第五层：元进化（Meta-Evolution）

> 进化协议本身也是可以进化的。

#### 5.1 元进化触发条件

```
- 进化协议拒绝了一次重构，但后续证明那次重构是必要的
  → 协议的"不该进化"标准需要校准

- 连续两次重构都没通过验证
  → 协议的决策逻辑可能有问题

- 双轨验证通过但上线后用户反馈变差
  → 协议的验证标准不够严格

- 同一个类型的翻译问题在不同语言对中反复出现
  → 协议缺少对跨语言模式的感知

- 技能长期没有进化，不是因为没问题，而是信号未被感知
  → 信号敏感度需要调整
```

#### 5.2 元进化的执行方式

```
① 发现元进化信号 → 记录但不立即行动
② 积累到 ≥3 个同类信号 → 提出元进化方案
③ 在下一次框架进化中，附带给元进化方案开一个"观察窗口"
   （不直接改协议，但在进化日志中标注"如果新协议会怎么判断"）
④ 经过至少两次框架进化周期的对比观察 → 决定是否正式修改元协议
⑤ 元协议修改 → 更新技能模板（影响到后面所有技能的制作）
```
